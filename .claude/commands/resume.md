恢复之前的工作状态并准备继续开发。$ARGUMENTS

# 执行步骤

## 1. 读取 Progress.md

读取项目根目录的 `progress.md`，重点关注：

- 「上次会话快照」：上次做到哪了、下一步是什么
- 「未完成」列表：待办事项全貌
- 「当前状态」：架构和环境的最新信息

## 2. 查看最近 Git 提交

运行以下命令，了解最近的代码变更：

```bash
git log --oneline -10
```

如果最近有提交，查看最新提交的详细内容：

```bash
git log -1 --stat
```

检查是否有未提交的变更：

```bash
git status
git diff --stat
```

## 3. 综合分析

将 Progress.md 的记录与 git 历史进行交叉对照：

- Progress.md 记录的「下一步」是否已被后续提交完成？
- 是否有 Progress.md 未记录的新提交？
- 是否有未提交的变更需要处理？

## 4. 输出恢复摘要

用以下格式向用户汇报：

```
## 工作状态恢复

**上次进度**：[一句话概括上次做到哪了]

**最近变更**（git log）：
- [commit 1 简述]
- [commit 2 简述]

**待处理**：
- [ ] [优先级最高的下一步]
- [ ] [其他待办]

**未提交变更**：[有/无，如有则简述]
```

## 5. 询问用户

向用户确认接下来要做什么：

- 如果有明确的「下一步」，提议继续该任务
- 如果用户通过 $ARGUMENTS 指定了任务，直接开始
- 否则列出待办选项，让用户选择

# 注意事项

- 恢复摘要要简洁，不要大段复述 progress.md 的全部内容
- 重点是让用户快速回忆上下文并进入工作状态
- 如果 progress.md 中没有「上次会话快照」section，仅基于「未完成」列表和 git log 汇报
