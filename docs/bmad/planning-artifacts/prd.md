# Product Requirements Document - wenexus

**Author:** xiaohui
**Date:** 2026-03-15

## Executive Summary

WeNexus 是一个 AI 驱动的多元视角内容平台，围绕人类社会的经典争议问题（彩礼、内卷、AI 取代人类、阶层固化等），通过结构化多视角呈现，让用户自己拼出事情的全貌。

**核心问题**：现有内容平台（知乎、微博、ChatGPT）在争议话题上只提供局部视角——单一作者的"全面解析"、刻意中立的折中答案、或算法驱动的极化内容。用户看到的是"我方的真相"，且多数人不知道自己只看到了局部。

**产品架构**：双层内容体系。表层是小红书式观点卡片信息流，每个话题由 5-8 张可滑动的图文卡片组成——首屏是钩子卡片（一句让你停下来的话 + 一个你没想到的角度），后续卡片逐层展开深度分析，构成完整的认知旅程；深层是 AI 专家圆桌讨论，多位 AI 专家进行线程式辩论，用户可挂机围观或接管主持，讨论成果一键转化为文章、决策清单等产出物。

**目标用户**：任何面对争议话题的普通人（覆盖全生命周期议题）。核心驱动力是自利——"我不想被蒙在鼓里"。进阶用户进入圆桌讨论，用理解去行动。

**商业模式**：会员积分制。免费用户浏览观点卡片，付费会员使用圆桌讨论和产出物生成。AI Agent 在外部平台自动运营内容导流，实现低成本冷启动。

**技术栈**：Next.js 15 + React 19 + TailwindCSS 4 + FastAPI + Drizzle ORM + LangChain/LangGraph，部署于 Cloudflare Workers。

### What Makes This Special

1. **全貌呈现，非单一解析**：不是一个人讲"来龙去脉"，而是让不同立场的专家各自讲自己看到的部分。比任何单一视角的"全面解析"都更接近真相。
2. **经典问题，持久资产**：聚焦不会过期的争议话题，每条内容都是长期资产。AI 批量生产能力使 50 个种子话题可扩展至 10000+。
3. **认知惊喜驱动，非情绪刺激**：制造"啊，原来如此"的好奇心，而非制造愤怒。内容消费体验，认知提升是副产品。
4. **AIGC 全链路**：AI 生成内容 → AI 渲染卡片（JSX 直出）→ AI Agent 分发 → AI 圆桌讨论，几乎零人力成本运营。

## Project Classification

- **项目类型**：Web App（Next.js SPA，响应式设计）
- **领域**：AI 驱动内容平台（多元视角认知升级）
- **复杂度**：中高（AI 内容生产引擎 + 多 Agent 圆桌系统 + AIGC 渲染 + 外部平台分发，无强合规要求）
- **项目状态**：Brownfield（已有技术栈、领域架构、32 个用户故事）

## Success Criteria

### User Success

**北极星指标：认知惊喜率（Cognitive Surprise Rate）**

- 定义：用户浏览观点卡片后产生"想看更多"行为的比例
- 度量：点击进入话题详情 / 浏览同话题下一张卡片的转化率
- 目标：> 40%（对比小红书平均内容互动率约 5-10%，WeNexus 因内容精准命中认知盲区应显著高于常规内容）

**Aha Moment**：用户首次在同一话题下浏览 3+ 个不同立场的观点卡片。

**用户成功阶梯：**

| 阶段 | 用户行为 | 成功信号 |
|------|---------|---------|
| 触达 | 在外部平台看到观点卡片 | 点击率 > 2% |
| 激活 | 进入 WeNexus 浏览同话题多视角 | 单话题浏览 3+ 卡片率 > 30% |
| 留存 | 次日/次周回访 | 次周留存 > 20% |
| 深度 | 进入圆桌讨论 | 卡片→圆桌转化率 > 5% |
| 付费 | 购买会员 | 注册→付费转化率 > 2% |

## Product Scope & Development Phases

### 架构核心洞察

> **圆桌讨论引擎 = 内容生产引擎。** 圆桌讨论在无用户输入时以 autopilot 模式运行，产出的默认内容即为信息流观点卡片。用户 kick in 后变为交互式圆桌。这不是两个模块，是同一个引擎的两种模式。

### MVP Strategy & Philosophy

**MVP 类型**：技术验证型 MVP（Problem-Solving MVP）
**核心验证假设**：AI 能生成让用户产生"认知惊喜"的多视角内容，并通过外部分发吸引用户。
**资源现实**：创始人 1 人 + AI 辅助开发，必须极致聚焦。
**核心竞争力**：大模型高质量内容产出能力是产品的核心壁垒，需重点攻克。

### MVP Feature Set (Phase 1) — 按开发优先级排序

**P0：圆桌讨论引擎（= 内容生产引擎）⭐ 核心竞争力**

- 求真者事实打底 + 多 AI 专家线程辩论
- Autopilot 模式：无用户输入时自动产出结构化多视角内容
- Interactive 模式：用户可挂机围观或接管主持
- Verify Agent 自动质量验证
- 产出物生成（文章、决策清单等）
- **大模型高质量产出是产品的核心壁垒，需重点投入 prompt engineering、专家人设设计、辩论引导策略**

**P1：观点卡片展示**

- 将圆桌引擎 autopilot 产出的内容渲染为可滑动图文卡组（钩子卡片 + 展开卡片，AIGC 直出 JSX）
- 话题维度的信息流浏览
- 卡片分享功能（自带品牌水印）
- SEO 友好的 SSR 页面

**P2：外部平台 Agent 分发**

- xhs 等工具自动发布到小红书
- 分发调度 + 监控面板
- 导流链路（外部 → WeNexus）

**P3：会员付费系统**

- 免费层（卡片浏览）+ 付费层（交互式圆桌 + 产出物）
- 会员积分制

### MVP 接口预留（架构完备原则）

> **核心原则：MVP 阶段不做的功能，接口需要留好，实现可以简单，架构一样完备。**

| 功能 | MVP 状态 | 接口预留 |
|------|---------|---------|
| 用户社交功能 | 不实现 | 用户关系模型、关注/粉丝 API 接口预留 |
| 评论系统 | 不实现 | 内容互动数据模型、评论 API 接口预留 |
| 内容创作者工具 | 不实现 | 创作者角色权限、内容生产 API 接口预留 |
| 多语言内容 | 不实现 | i18n 内容模型、locale 路由已有（next-intl） |
| 个性化推荐 | 不实现 | 用户行为埋点、推荐 API 接口预留 |

**Phase 2（Growth）：**

- 多平台 Agent 矩阵分发（小红书、微博、公众号、B站等）
- 话题库扩展至 10000+（批量 autopilot 产出）

**Phase 3（Expansion）：**

### Vision (Future)

- **全球认知升级信息中心**：多语言覆盖，成为争议话题"看全貌"的第一选择
- **内容创作者生态**：开放多余产能和工具给创作者
- **Nexus 人社区**：从内容平台进化为认知共同体，线上认知到线下连接
- **全球人类认知升级的基础设施**

### Risk Mitigation Strategy

**技术风险（最高）：** LLM 能否产出高质量多视角辩论？

- 这是核心竞争力所在，需重点攻克而非回避
- 缓解：前 50 个话题密集迭代 prompt engineering 和专家人设；Verify Agent 自动过滤；持续优化辩论引导策略

**市场风险：** 用户真的想看"全貌"吗？

- 缓解：通过 Agent 在小红书分发验证——观点卡片的点击率和分享率是最快的市场信号

**资源风险：** 1 人开发，能否在 3 个月内交付 MVP？

- 缓解：P0-P3 串行推进，P0 是一切基础；AI 辅助开发提效；架构完备但实现从简

### Journey 1: 小陈的认知觉醒（主要用户 - 成功路径）

**人物**：陈浩，27 岁，互联网公司产品经理，刚和女友因为彩礼问题大吵了一架。

**Opening Scene**：
晚上 11 点，陈浩躺在床上刷小红书，心里还在生气。他觉得女友家要 28 万彩礼不合理，但又说不清楚为什么心里发堵。刷到一张观点卡片——标题是"彩礼的本质不是买卖，是两个家庭对未来不确定性的对冲"。他停下了拇指。

**Rising Action**：
这个角度他从没想过。他点进 WeNexus，看到"彩礼"话题下有多张长文本图文卡片，每张卡片融合了不同维度的分析和数据引用：经济学家讲资源转移、社会学家讲文化惯性、女性主义学者讲身体自主权、农村研究者讲返乡青年的现实困境……每张卡片都让他停下来想一想。

**Climax**：
他看到一张卡片中的核心论点："对于女方父母来说，彩礼是女儿离开原生家庭后、唯一能确认女婿家庭重视程度的信号。这不是贪婪，是不安全感。"他突然理解了女友妈妈为什么那么坚持。不是被说服了，而是**第一次看到了对方的逻辑**。

**Resolution**：
陈浩注册了 WeNexus，又看了"婚前同居""原生家庭"两个话题。第二天，他主动和女友说："我昨晚看到一些东西，我觉得我之前只从自己的角度想了。"

> **揭示的能力需求**：观点卡片展示（长文本图文）、话题多视角浏览、外部平台（小红书）分发与导流、用户注册

### Journey 2: 王沛然的深度行动（进阶用户 - 圆桌讨论路径）

**人物**：王沛然，32 岁，中学教师，需要说服年迈的父亲停止开车。

**Opening Scene**：
沛然已经是 WeNexus 的老用户了，平时会浏览观点卡片。这次她有一个真实的难题：父亲 70 岁了，反应变慢但坚持开车，说"不开车等于被关在家里"。

**Rising Action**：
她在 WeNexus 找到"老年人驾驶与家庭关怀"话题，但这次她需要更深入。她点击"进入圆桌讨论"，选择付费开启。求真者先呈现数据：70 岁以上驾驶事故率、各国政策对比、替代出行方案成本。然后老年心理学专家、交通安全专家、家庭关系顾问开始辩论。

**Climax**：
沛然从"挂机围观"切换到"接管主持"，追问："如果老人的核心需求是'不被困在家里'，有没有不剥夺驾驶权的方案？"AI 专家们围绕她的具体问题展开讨论，提出了"渐进式过渡方案"。她点击"生成沟通策略"，得到一份包含开场白、核心论点和预案的清单。

**Resolution**：
周末回家，沛然没有说"你不能再开车了"，而是说"爸，我查了一些东西，想和你商量一个方案"。父亲第一次没有发脾气，因为他感受到的不是控制，而是理解。

> **揭示的能力需求**：圆桌讨论（求真者 + 多专家辩论）、挂机/主持模式切换、产出物生成（沟通策略）、付费会员权限验证

### Journey 3: 内容生产与自动验证（系统 + 创始人异常介入）

**人物**：Content Agent（生成）+ Verify Agent（验证）+ xiaohui（异常介入）

**Opening Scene**：
凌晨 2 点，Content Agent 开始批量生产。今天的任务是"35 岁危机"话题——Agent 生成了一组图文卡片，每张卡片是一个完整的长文本图文展示，融合了多个维度的分析、数据引用和专家观点。

**Rising Action**：
生成完成后，Verify Agent 自动接管。它按照预设的质量标准逐一验证：论证是否有数据支撑、视角是否覆盖足够多元、表述是否有明显偏见或事实错误、图文排版是否符合设计规范。5 张卡片中，4 张通过验证，1 张被标记为"论证薄弱——缺少具体劳动力市场数据"。

**Climax**：
通过的 4 张卡片自动进入线上话题库和分发队列。被标记的 1 张触发通知给 xiaohui。xiaohui 查看后选择让 Content Agent 重新生成并补充具体数据，Verify Agent 再次验证通过。

**Resolution**：
整个人工介入只花了 5 分钟。日常状态下，大部分内容完全自动完成，xiaohui 只处理异常。

> **揭示的能力需求**：Content Agent 内容生产流水线、Verify Agent 自动质量验证、异常通知与人工介入工作流、内容状态管理（生成中/验证中/通过/异常/重新生成）

### Journey 4: AI Agent 自动分发与监控（系统运维 - 自动化路径）

**人物**：自动化系统 + xiaohui（异常时介入）

**Opening Scene**：
每天下午 2 点和晚上 8 点，AI Agent 自动从分发队列中取出通过验证的观点卡片，通过 xhs 等平台工具自动发布到小红书账号。发布内容包括：卡片图片 + 精炼文案 + WeNexus 导流链接。

**Rising Action**：
系统监控面板显示今日发布情况：发布 3 条，其中 1 条获得 500+ 点赞，1 条表现平平，1 条被小红书限流（可能触发了敏感词）。监控面板自动标记限流内容为"异常"。

**Climax**：
xiaohui 收到异常通知，查看被限流的内容，发现是"两党对立"话题的卡片。他在监控面板上将这条标记为"暂停分发"，并调整该话题的分发策略。

**Resolution**：
监控面板记录了本周的分发数据：总发布 21 条，总曝光 15 万，导流 WeNexus 访问 800 次，注册 45 人。这些数据验证了分发链路的可行性。

> **揭示的能力需求**：自动化分发调度、xhs 等平台工具集成、分发监控面板（发布状态/数据/异常）、异常告警通知、分发策略配置

## Domain-Specific Requirements

**领域定位**：AI 驱动内容平台，中高复杂度，无强合规要求。

**关键领域约束：**

- **内容安全红线**：短期内不设内容限制，优先验证产品可行性。产品存活比自我限制更重要。
- **LLM 供应商**：通过 OpenRouter 统一接口，支持多供应商切换（详见 FR46-47）。
- **外部平台合规**：小红书等平台有各自的内容审核规则，逐个案例处理，不预设系统级红线。

## Innovation & Novel Patterns

### Detected Innovation Areas

1. **AIGC 全链路自动化**：Content Agent 生成 → Verify Agent 验证 → 自动上线 → AI Agent 分发。内容行业首个从生产到分发的全自动化闭环。
2. **多 Agent 结构化辩论**：多个有立场的 AI 专家进行线程式对话、互相引用和反驳，区别于单一 LLM 的"客观答案"模式。LLM 应用新范式。
3. **经典问题作为持久资产**：反行业常态（追热点），聚焦不过期的争议话题，改变内容平台的经济模型。

### Validation Approach

Phase 1 的 50 个话题直接验证全部创新假设：AI 生成质量、Verify Agent 有效性、用户"认知惊喜"吸引力、分发链路可行性。

## Web App Specific Requirements

### Project-Type Overview

WeNexus 是一个内容消费型 Web App，短期作为产品验证平台，长期会有 App 版本。网站本质上是运营和验证的前沿阵地。

### Technical Architecture Considerations

**浏览器支持**：仅现代浏览器（Chrome/Safari/Edge/Firefox 最新 2 个版本），不支持 IE。

**SEO 策略**：

- 观点卡片页面需要 SSR/SSG 以被搜索引擎收录（Next.js App Router Server Components 天然支持）
- 话题列表页、卡片详情页为 SEO 重点页面
- 圆桌讨论页面不需要 SEO（需要登录/付费）
- 结构化数据（JSON-LD）标记话题和观点内容

**实时能力（混合架构）**：

- **WebSocket**：圆桌讨论实时通信（已实现 ConnectionManager + auto-reconnect + polling fallback）
- **SSE**：AI 流式输出（已实现 AI SDK `streamText()` + `useChat()`）
- **待补**：OpenRouter 流式调用，实现圆桌 AI 专家回复逐字流式输出

**响应式设计**：

- Mobile-first 设计（观点卡片主要在手机端消费）
- 卡片信息流需要流畅的移动端滚动体验
- 圆桌讨论在桌面端有更好的多线程展示

**性能目标**：详见 Non-Functional Requirements 性能节（NFR1-NFR6）。

### Implementation Considerations

**部署约束**：

- Cloudflare Workers 不支持 Node.js 原生模块（shiki/prettier/yaml/acorn 已 stub）
- 构建必须设置 `CF_BUILD=true`
- next.config 中间件栈顺序不可更改：withMDX → withNextIntl → withBundleAnalyzer

**国际化**：

- next-intl 4.3.4 已集成，locale-based routing（/en/\*, /zh/\*）
- MVP 仅生产中文内容，但架构面向全球用户

**LLM 供应商抽象**（详见 FR46-47、NFR13/15）：

- 当前通过 OpenRouter 统一接口，已有 `util/llm.py` 抽象层
- 待补：OpenRouter 流式调用，支持圆桌 AI 专家回复逐字输出

## Functional Requirements

### 话题与内容生产

- **FR1**: 系统可以存储和管理经典争议话题库（含话题元数据、分类、标签）
- **FR2**: 系统可以将一个经典争议话题拆解为多个争议维度
- **FR3**: 系统可以为每个争议维度生成不同立场的 AI 专家视角
- **FR4**: 系统可以管理内容的生命周期状态（生成中 → 验证中 → 通过 → 异常 → 重新生成）
- **FR5**: 系统可以批量离线生产话题内容（吞吐量标准见 NFR11）
- **FR6**: 系统可以增量在线生成单个话题内容
- **FR7**: 系统可以展示话题之间的关联关系，用户可以沿关联路径浏览

### 圆桌讨论引擎

- **FR8**: 系统可以以 Autopilot 模式（无用户输入）自动产出结构化多视角讨论内容
- **FR9**: 求真者（Fact Checker）可以为每次讨论提供事实数据基础
- **FR10**: 多位 AI 专家可以进行线程式辩论，明确引用和回应彼此观点
- **FR11**: 用户可以以「挂机围观」模式被动观看 AI 专家实时讨论
- **FR12**: 用户可以以「接管主持」模式输入问题引导讨论方向
- **FR13**: 用户可以在挂机围观与接管主持模式之间自由切换
- **FR14**: AI 专家回复支持实时渐进式呈现
- **FR15**: 用户可以回到之前的圆桌讨论继续对话

### 观点卡片消费

- **FR16**: 用户可以在话题维度的信息流中浏览观点卡片
- **FR17**: 用户可以在同一话题下浏览多个不同立场的观点卡片
- **FR18**: 系统可以将 AI 产出的内容渲染为长文本图文卡片（AIGC 直出 JSX）
- **FR19**: 用户可以分享观点卡片（分享内容自带品牌水印）
- **FR20**: 观点卡片页面支持 SEO 友好的服务端渲染
- **FR21**: 观点卡片支持移动端优先的响应式浏览体验
- **FR22**: 用户可以按生命阶段或主题分类浏览话题
- **FR23**: 免费用户可以预览圆桌讨论的摘要或精彩片段

### 内容搜索与发现

- **FR24**: 用户可以通过自然语言搜索话题和关联的观点卡片
- **FR25**: 用户可以在搜索结果基础上连续发起追问（多轮对话式搜索）
- **FR26**: 用户可以从搜索结果无缝进入相关话题的圆桌讨论
- **FR27**: 系统可以在首页为用户推荐相关话题和观点卡片

### 产出物生成

- **FR28**: 用户可以指定产出物类型，系统根据圆桌讨论内容生成对应格式的产出物
- **FR29**: 用户可以下载或分享生成的产出物

### 外部分发与导流

- **FR30**: AI Agent 可以自动在外部平台（小红书等）发布观点卡片内容
- **FR31**: 系统可以按预设时间调度自动分发任务
- **FR32**: 外部平台发布内容包含 WeNexus 导流链接
- **FR33**: 管理员可以查看分发监控面板（发布状态、曝光量、导流量、异常标记）
- **FR34**: 管理员可以标记异常内容并调整分发策略（暂停/恢复/修改）
- **FR35**: 系统可以对分发异常自动触发告警通知

### 用户管理与付费

- **FR36**: 用户可以注册和登录 WeNexus
- **FR37**: 免费用户可以无限浏览观点卡片信息流
- **FR38**: 付费会员可以使用交互式圆桌讨论功能
- **FR39**: 付费会员可以生成和获取产出物
- **FR40**: 系统支持会员积分制（积分获取、消耗、查询）
- **FR41**: 系统支持付费购买会员/积分

### 质量保障与运维

- **FR42**: Verify Agent 可以自动验证 AI 生成内容的质量（论证完整性、视角多元性、事实准确性、排版规范）
- **FR43**: 通过验证的内容自动进入线上话题库和分发队列
- **FR44**: 未通过验证的内容自动标记为异常并触发通知
- **FR45**: 管理员可以对异常内容进行人工介入处理（重新生成/编辑/放弃）
- **FR46**: 系统通过统一接口（OpenRouter）支持多种 LLM 供应商切换
- **FR47**: 系统支持对不同功能模块配置不同 LLM 模型
- **FR48**: 系统可以采集和记录用户关键行为数据（浏览、点击、停留、分享、转化）

## Non-Functional Requirements

### 性能

- **NFR1**: 观点卡片首屏加载时间 < 2s（Cloudflare Workers 边缘部署 + SSR）
- **NFR2**: AI 专家单轮流式回复首 token 延迟 < 2s
- **NFR3**: 圆桌讨论全程流畅无明显卡顿，用户无需等待超过 5s
- **NFR4**: AIGC JSX 卡片渲染成功率 > 95%，跨现代浏览器兼容
- **NFR5**: 自然语言搜索返回结果 < 3s
- **NFR6**: 移动端卡片信息流滚动体验流畅（60fps）

### 可靠性

- **NFR7**: 系统整体可用性 > 99.5%
- **NFR8**: 内容生产流水线（Autopilot + Verify Agent）支持断点续传，单个话题失败不阻塞批量任务
- **NFR9**: 外部平台分发任务失败时自动重试，并记录失败原因
- **NFR10**: WebSocket 断连后自动重连，支持 polling 降级

### 可扩展性

- **NFR11**: 离线批量内容生产吞吐量 > 100 话题/天
- **NFR12**: 话题库从 50 扩展至 10000+ 时，卡片浏览性能无明显退化
- **NFR13**: LLM 供应商切换不影响线上服务，支持热切换
- **NFR14**: 系统架构支持未来新增外部分发平台，无需核心代码重构

### 集成

- **NFR15**: 通过 OpenRouter 统一接口对接 LLM 供应商，接口超时、限流有降级处理
- **NFR16**: 外部平台（小红书等）分发集成支持平台 API 变更后的快速适配
- **NFR17**: 支付系统集成满足基本交易安全（HTTPS、支付令牌化）
