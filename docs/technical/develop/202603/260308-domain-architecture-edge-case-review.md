## 一、领域事件契约缺口（7 项）

### 1.1 `DISCUSSION_READY.readiness` 无范围约束

- **触发条件**：readiness 为负数、NaN 或超出预期范围
- **后果**：Deliverable 域误判就绪阈值，过早生成或永不触发
- **建议**：约束为 `0..1` 或 `0..100`，并定义阈值语义

### 1.2 `CONSENSUS_UPDATED.level` 无范围和单位定义

- **触发条件**：level 值无明确范围或含义
- **后果**：Discovery 展示无意义或错误的共识度指标
- **建议**：指定范围（0-100? 0-1?）及各边界含义

### 1.3 `FEED_CONTENT_DISTILLED.cards` 可为空数组

- **触发条件**：蒸馏事件触发但 cards 为 `[]`
- **后果**：Discovery 收到事件但无内容可渲染
- **建议**：使用非空元组类型 `[string, ...string[]]` 或定义空数组语义

### 1.4 无失败/补偿事件

- **触发条件**：蒸馏失败、专家生成失败等异常场景
- **后果**：域间管道中断但无法感知，静默失败
- **建议**：增加 `DISTILLATION_FAILED`、`EXPERT_GENERATION_FAILED` 等事件

### 1.5 事件缺少幂等性标识

- **触发条件**：事件重复投递
- **后果**：重复处理导致重复卡片、状态损坏
- **建议**：在事件基类中增加 `eventId: string` 和 `timestamp: number`

### 1.6 无话题删除/归档事件

- **触发条件**：话题被删除或归档
- **后果**：Roundtable 残留孤立会话，Discovery 残留过期卡片
- **建议**：增加 `TOPIC_ARCHIVED` 事件并定义级联清理策略

### 1.7 `USER_ACTION.context` 类型过于宽松

- **触发条件**：`Record<string, unknown>` 无 schema 校验
- **后果**：Identity 域收到非结构化数据，偏好积累不可靠
- **建议**：按 action 类型定义判别联合类型（discriminated union）

## 二、ACL 接口契约缺口（4 项）

### 2.1 `getDiscussionContext(topicId)` 无空值契约

- **触发条件**：topicId 不存在
- **后果**：Deliverable 域崩溃或生成空交付物
- **建议**：定义返回类型 `DiscussionContext | null`

### 2.2 `getDistilledFeedContent(topicId)` 无初始状态处理

- **触发条件**：话题尚未进行任何蒸馏
- **后果**：Discovery 信息流对该话题显示空白或报错
- **建议**：定义 `FeedContent | null` 及空状态 UI 策略

### 2.3 `getUserPreference(userId)` 无新用户处理

- **触发条件**：新注册用户无偏好记录
- **后果**：Roundtable 个性化逻辑出现不可预测的默认行为
- **建议**：定义默认偏好 shape 或 fallback 策略

### 2.4 所有 ACL 接口缺少返回类型和错误类型

- **触发条件**：不同域的实现者独立理解接口
- **后果**：返回数据结构不兼容，集成阶段才暴露问题
- **建议**：在契约表中补充返回类型和错误类型

## 三、结构性矛盾（4 项）

### 3.1 3 人配置违反"一域一 Owner"原则

- **位置**：§4.2 工程师 B 同时拥有 Discovery 和 Deliverable 两个域
- **矛盾**：§4.1 明确规定"一个域 = 一个 Owner"
- **建议**：在 3 人配置中显式声明此例外，或在该规模下合并为一个临时域

### 3.2 Shared Kernel 在 3 人团队中无明确 Owner

- **触发条件**：LLM Gateway 需要修改时无人负责审查
- **后果**：跨域回归风险
- **建议**：显式指定 Shared Kernel 归属（如"工程师 A 拥有 SK，B/C 通过 PR 贡献"）

### 3.3 M1 里程碑存在循环依赖

- **位置**：§7.1 M1 包含 US-26（观点卡片生成），但需要 Roundtable 讨论数据
- **后果**：M1 中 Roundtable 尚未稳定，Deliverable 无法完成集成
- **建议**：补充依赖说明：M1 的 US-26 使用 stub 数据，或推迟到 M2

### 3.4 Shared Kernel 7 项能力仅 2 项有实现引用

- **触发条件**：仅 LLM Gateway 和 Design System 有文件引用
- **后果**：工程师假设能力已存在，实际仍为 planned
- **建议**：标注各能力当前状态（已有 / 桩 / 计划中）

## 四、流程/策略缺失（5 项）

### 4.1 讨论永不就绪的死锁场景

- **触发条件**：Roundtable 讨论从不达到 `DiscussionReady` 状态
- **后果**：Deliverable 管线永久阻塞，用户无任何产出
- **建议**：定义超时策略或最低阈值策略

### 4.2 Content Distillation 模式选择规则未定义

- **触发条件**：离线批量 vs 在线实时两种模式无明确触发条件
- **后果**：同一话题触发重复蒸馏或遗漏蒸馏
- **建议**：定义决策规则（如：讨论结束 → 离线；新话题无卡片 → 在线）

### 4.3 双重交付的部分失败处理

- **触发条件**：对用户交付成功但对信息流交付失败，或反之
- **后果**：系统状态不一致
- **建议**：定义原子性策略（独立 best-effort 各自重试，或 saga 补偿）

### 4.4 私密话题的内容泄露风险 ⚠️ 最高优先级

- **触发条件**：`TOPIC_CREATED { isPrivate: true }` 经 Roundtable 讨论后，Deliverable 域触发 `FEED_CONTENT_DISTILLED` 回流到公开信息流
- **后果**：**私密讨论内容泄露到公开 Discovery 信息流**
- **建议**：在事件契约中明确：私密话题的 `FEED_CONTENT_DISTILLED` 事件必须被 suppress 或标记为 `visibility: 'private'`

### 4.5 专家生成失败无恢复路径

- **触发条件**：`TopicCreated` 事件触发专家生成，但 LLM 调用失败
- **后果**：用户创建了话题但圆桌无专家，讨论无法开始
- **建议**：定义 `ExpertGenerationFailed` 事件或重试策略

## 优先处理建议

| 优先级 | 编号 | 问题 | 原因 |
|--------|------|------|------|
| **P0** | 4.4 | 私密话题内容泄露 | 隐私安全，不可上线前遗留 |
| **P0** | 1.4 | 无失败事件 | 影响全域容错能力 |
| **P1** | 2.1-2.4 | ACL 接口契约 | 影响域间集成的正确性 |
| **P1** | 4.1 | 讨论死锁 | 影响核心用户体验 |
| **P2** | 其余 | 值约束、幂等性等 | 影响长期稳定性 |
