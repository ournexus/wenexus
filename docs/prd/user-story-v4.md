## 一、产品定位

### 1.1 产品使命

**促进人与人之间的相互理解，打造更加包容、更加和谐的社会。**

WeNexus 的核心信念：社会争议的本质往往是"不同视角的人看到了不同的真相"。我们不追求强制共识，而是帮助用户理解"为什么有人会这样想"。

### 1.2 一句话描述

WeNexus 是一个 **AI 多元视角内容平台**，通过预生成的图文卡片快速呈现争议话题的多方观点，用户可进一步进入圆桌讨论深度探索。

### 1.3 双层内容架构

WeNexus 采用"表层轻量消费 + 深层深度互动"的双层架构（完整图示见「四、内容架构」）：

- **表层**：预生成的观点卡片信息流（小红书风格），低门槛获客、快速消费、社交传播
- **深层**：圆桌讨论过程（可选链路），深度互动、个性化服务、付费转化

### 1.4 核心价值主张

- **对用户**: 快速了解争议话题的多元视角，促进理解而非站队
- **对创作者**: 获得有深度、有冲突、可直接使用的内容素材
- **对决策者**: 模拟"专家会诊"，辅助复杂决策
- **对投资人**: 轻量消费（获客）+ 深度服务（变现）的双层商业模式

### 1.5 产品隐喻

| 角色 | 隐喻 |
|------|------|
| 用户 | 圆桌会议主持人 / 观察者 / 参与者 |
| AI 副主持人 | 会议助理（挂机模式下代理用户） |
| AI 专家 | 圆桌嘉宾（代表不同立场和视角） |
| 平台 | 理解的桥梁 / 多元视角的聚合器 |
| 话题 | 社会议题 / 个人困境 |
| 产出物 | 理解地图 / 内容素材 / 决策建议 |

### 1.6 核心设计理念

1. **促进理解优先**：不追求强制共识，而是帮助用户理解"为什么有人会这样想"
2. **角色公平代表**：确保每种立场都有专家公平发声，避免预设"正确答案"
3. **双层体验**：表层轻量消费（观点卡片），深层深度互动（圆桌讨论）
4. **事实先行**：每场讨论由"求真者"开场，确保基于事实辩论
5. **交付导向**：讨论流程围绕最终交付物设计

## 三、核心机制

> 本章集中定义 WeNexus 的核心产品机制。用户故事（第六章）和交互设计（第七章）会引用本章概念，不再重复解释。

### 3.1 专家角色设计原则

为了实现"促进理解"的使命，专家角色设计遵循以下原则：

| 原则 | 说明 | 示例 |
|------|------|------|
| **立场多元** | 每个争议话题至少有 2-3 种截然不同的立场 | 彩礼问题：传统派（王阿姨）vs 现代派（高中生）vs 分析派（费孝通） |
| **公平发声** | 每种立场的专家有同等的发言质量和深度 | 不能让"正方"说三段论，"反方"只说情绪化的话 |
| **避免稻草人** | 每个专家真正代表其立场的最佳论证 | 保守派不是"顽固不化"，而是有其合理关切 |
| **展示理由** | 专家不仅表达观点，还解释"为什么这样想" | "作为企业主，我的顾虑是..." |
| **承认不确定性** | 专家可以表达不确定，而非总是斩钉截铁 | "这个问题我也没有答案，但我的倾向是..." |

### 3.2 求真者开场机制

每场圆桌讨论开场流程：

1. **求真者（Fact Checker）自动第一个发言**，执行 Deep Research
2. 搜索最新新闻报道、官方数据、权威研究
3. 输出结构化的事实陈述（关键数据、最新动态、争议焦点）
4. 所有信息标注来源，确保可追溯
5. 其他专家基于求真者提供的事实展开讨论

**求真者的核心职责**：

- 信息搜集：利用实时搜索能力获取最新、最全的事实信息
- 中立陈述：只陈述事实，不做价值判断
- 来源标注：所有信息标明出处，确保可追溯
- 争议识别：指出事实层面的分歧点，但不站队

### 3.3 线程对话机制

专家发言时明确引用其他专家的观点，形成有来有往的真正对话：

| 规则 | 说明 | 示例 |
|------|------|------|
| 必须@回应对象 | 每次发言明确回应谁 | "@鲁迅，你说的..." |
| 引用具体观点 | 引用对方的原话片段 | 针对"xxx"这个说法... |
| 表明态度 | 认同/反驳/补充/质疑 | 我不同意/我想补充/我有疑问 |
| 推进讨论 | 不能只否定，要有建设性 | 我认为更准确的说法是... |

### 3.4 挂机/接管模式

WeNexus 的圆桌讨论支持两种操控模式（交互细节见「7.2 挂机/接管状态栏」）：

- **挂机模式（默认）**：AI 副主持人自动推进讨论——智能选择发言者、发现浅尝辄止时自动追问、平衡发言机会。用户可随时点击「接管」介入。
- **接管模式（手动）**：用户直接控制讨论——点名特定专家发言、要求专家互相反驳、追问某个观点、发起全体投票。用户可随时点击「恢复挂机」。

### 3.5 交付导向的讨论引导

用户创建话题时选择期望的交付物，AI 主持人据此引导讨论方向：

| 交付物类型 | AI 引导策略 | 收敛时机 |
|-----------|------------|---------|
| 共识报告 | 覆盖全面的观点维度 | 各方观点充分表达后 |
| 公众号文章 | 挖掘金句和案例故事 | 形成清晰叙事主线后 |
| 视频/播客脚本 | 制造戏剧性冲突和高潮 | 有足够"名场面"后 |
| 决策清单 | 聚焦用户具体场景 | 形成可执行建议后 |

### 3.6 理解地图（P1）

> 该概念用于 P1 阶段替代基线阶段的「共识度」指标，目的是更贴近"促进理解"的产品使命。

传统的"共识度"指标暗示了"达成共识是好的"，这与我们"促进理解"的使命可能冲突。因此，我们引入"理解地图"概念：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           理解地图                                           │
│                                                                             │
│   维度1: 事实共识度 ████████░░ 80%  ← 双方对事实本身是否一致                   │
│   维度2: 立场理解度 ██████░░░░ 60%  ← 是否理解对方"为什么这样想"               │
│   维度3: 分歧根源   ████░░░░░░ 40%  ← 是否找到分歧的真正原因                   │
│   维度4: 可行妥协   ██░░░░░░░░ 20%  ← 是否存在双方可接受的方案                 │
│                                                                             │
│   💡 当前状态: "各方观点已充分表达，分歧根源逐渐清晰"                           │
│                                                                             │
│   与传统"共识度"的区别:                                                      │
│   • 不暗示"100%共识是好的"，而是展示"理解有多深"                               │
│   • 即使最终无法达成共识，用户也能理解"为什么有人会这样想"                      │
│   • 体现产品使命：促进理解，而非强制共识                                       │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 五、用户旅程

### 5.1 旅程阶段概览

```
首页（浏览卡片）→ 轻量对话（背景收集）→ 圆桌讨论（挂机/接管）→ 产出选择（生成交付物）
```

| 阶段 | 触点 | 用户行为 | 用户想法 | 用户情绪 |
|------|------|---------|---------|---------|
| 发现 | 首页卡片流 | 浏览观点卡片、被钩子吸引 | "这个观点有意思" | 好奇 → 期待 |

| 产出 | 生成内容 | 选择产出类型、导出分享 | "这内容直接能用" | 满足 → 惊喜 |

### 6.3 圆桌讨论 - 开场与对话机制

> 核心机制定义见「三、核心机制」，本节聚焦用户故事和验收标准。

#### US-07: 求真者事实打底

```
作为 用户
我想要 每场讨论开始前有专家先陈述客观事实
以便 确保后续讨论基于共同的事实认知
```

### 7.4 专家状态系统

| 状态 | 视觉表现 | 触发条件 |
|------|---------|---------|
| 默认/旁观 | 灰色边框，正常大小 | 未参与当前发言 |
| 正在发言 | 金色边框 + 脉冲动画，放大 | 当前正在输出内容 |
| 被点名/思考中 | 蓝色边框 + 加载动画 | 用户点名，AI 正在生成 |
| 强烈反对 | 红色边框 + 轻微抖动 | AI 判断观点冲突 |
| 表示认同 | 绿色边框 + 点头动画 | AI 判断观点一致 |
| 被静音 | 灰色 + 半透明，缩小到外圈 | 用户选择静音 |
| 想要发言 | 黄色边框 + 举手图标 | AI 判断有补充观点 |

```
- 背景色调变冷（蓝绿）
- 提示: "共识正在形成..."

共识下降时:
- 仪表盘数字变红，向下动画
- 专家彼此远离
- 背景色调变暖（红橙）
- 提示: "分歧加剧..."

新观点产生时:
- 仪表盘轻微震动
- 发言专家高亮
- 提示: "💡 新观点"

用户发言被回应时（参与者模式）:
- 专家头像转向用户
- 显示 "@你" 标识
- 提示: "专家正在回应你..."
```

```typescript
interface UserProfile {
  // 基础信息
  id: string;

  // 身份偏好（从轻量对话收集）
  preferredRole: 'student' | 'professional' | 'creator' | 'researcher' | 'curious';
  industry?: string;

  // 讨论偏好
  preferredExperts: string[];      // 喜欢的专家
  mutedExperts: string[];          // 静音的专家
  preferredDepth: 'quick' | 'deep'; // 讨论深度偏好
  preferredMode: 'host' | 'participant'; // 默认模式

  // 产出偏好
  preferredOutputFormats: ('report' | 'card' | 'checklist' | 'script' | 'social')[];

  // 历史数据
  topicsCreated: string[];         // 创建的话题
  topicsParticipated: string[];    // 参与的话题
  averageRounds: number;           // 平均讨论轮数
  outputsGenerated: number;        // 生成产出数
}
```
